# Experiment Tracking and Model Management

## Learning Objective
Students should understand the challenges of managing multiple machine learning experiments and how tools like MLflow solve these problems by tracking parameters, metrics, and models.

## What Is This Topic?
Experiment tracking is the process of saving all the metadata related to a machine learning training run. This includes the hyperparameters used (e.g., learning rate, number of trees), the resulting metrics (e.g., accuracy, MSE), and the final model file itself. 

## Why This Topic Matters
When developing models, data scientists often run dozens or hundreds of experiments to find the best performance. Without tracking, it is common to forget which combination of parameters resulted in the best model. It solves the "I had 95% accuracy yesterday, but I don't remember what code I ran" problem.

## Core Intuition
Imagine trying to bake the perfect chocolate chip cookie. You bake 20 batches, slightly changing the amount of sugar, baking time, and oven temperature each time. If you don't write down the recipe for each batch along with how good the cookies tasted, you won't be able to recreate the best batch. Experiment tracking is your recipe journal.

## Key Concepts
- **Run:** A single execution of training code.
- **Experiment:** A collection of runs for a specific project or task (e.g., "Customer Churn Prediction").
- **Parameters:** The inputs or settings for the run (e.g., `n_estimators=100`, `max_depth=5`).
- **Metrics:** The numerical outputs evaluated during or after training (e.g., `accuracy=0.85`, `loss=0.2`).
- **Artifacts:** Output files generated by the run, most importantly the trained model file (`.pkl`, `.h5`), but also plots or datasets.
- **Model Registry:** A centralized hub to manage the lifecycle of an ML model (e.g., transitioning a model from "Staging" to "Production").

## Step-by-Step Explanation (Using MLflow)
1. Import MLflow in your Python training script.
2. Set the experiment name (to group related runs).
3. Start an MLflow "run" block (`with mlflow.start_run():`).
4. Train your model.
5. Log your parameters (`mlflow.log_param()`).
6. Log your evaluation metrics (`mlflow.log_metric()`).
7. Log the trained model (`mlflow.sklearn.log_model()`).
8. Open the MLflow UI in your browser to view and compare all runs.

## Output / Result Interpretation
The primary output is the UI dashboard (often accessed via `localhost:5000`) where you can see a table of all runs, sort them by specific metrics, and select multiple runs to plot their metrics against each other visually.

## Real-World Uses
- A team of 5 data scientists working on the same project can log all their experiments to a shared central server. They can see what approaches others have tried to avoid duplicating work.
- An automated CI/CD pipeline trains a new model every week. It logs the new model to the registry, compares its accuracy against the current production model, and automatically promotes the new one if it is better.

## Advantages
- **Reproducibility:** You always know exactly what code and parameters created a specific model.
- **Comparability:** Easy visual comparison of hundreds of runs.
- **Governance:** Clear history of who trained what model and when it was promoted to production.

## Limitations
- **Overhead:** Adds slightly more code to training scripts.
- **Infrastructure:** Requires setting up a tracking server and database for team-wide use (though local use is very easy).

## Common Mistakes
- **Logging Everything:** Logging unnecessary data or too many metrics can make the UI cluttered and slow down training.
- **Inconsistent Naming:** If you log `learning_rate` in one run and `lr` in another, it becomes hard to compare them.
- **Forgetting to Log the Code Version:** Always track which git commit was used to run the experiment.
